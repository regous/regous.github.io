WPF：进度条刷新

​ 界面卡死非常影响用户体验。然而有时候任务的处理的确是需要一定的时间等待，对于比较确定的、短时间的等待，可以使用鼠标光标上的“忙”来提示用户后台正在任务。但是有时候需要长时的任务，这个时候如果有可能就要用到进度条。为在处理上的简单，程序员经常直接在按钮的点击响应函数（WPF 里是绑定的 ICommand）里进行任务的处理，这个时候进度条的更新就有了问题。

​ 如何解决这个问题呢？

测试控件

​ 实现一个 UserControl 使用测试进度的更新

ProgressBarUserControl.xaml

<UserControl x:Class="ProgressBarDemo.Views.ProgressBarUserControl"
             xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
             xmlns:d="http://schemas.microsoft.com/expression/blend/2008"
             xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
             xmlns:mc="http://schemas.openxmlformats.org/markup-compatibility/2006"
             mc:Ignorable="d" 
             d:DesignHeight="450" d:DesignWidth="800">
    <Grid Margin="10,10">
        <ProgressBar x:Name="progressbar" Height="20" Value="{Binding Path=Progress}"/>
    </Grid>
</UserControl>

            DependencyProperty.Register("Progress", typeof(double), typeof(ProgressBarUserControl), new PropertyMetadata(0.0));

        public void SetProgress(double progress)
        {
            Progress = progress;
        }

        public virtual void Start()
        {
            
        }
    }
}

DoEvents

​ 最简单的方法，在设置完进度之后，调用 System.Windows.Forms.Application.DoEvents 强制刷新控件。

namespace ProgressBarDemo.Views
{
    public class DoEventsUserControl : ProgressBarUserControl
    {
        public override void Start()
        {
            for (int i = 0; i <= 100; i++)
            {
                Progress = i;
                System.Windows.Forms.Application.DoEvents();
                System.Threading.Thread.Sleep(30);
            }
        }
    }
}

​ 对于 DoEvents 的作用，我不太了解，猜想可能是强制当前线程处理消息泵中的消息。因为是 Windows 是消息驱动的，而 UI 更新的消息等级不高，强制处理消息泵中的消息时，UI 就得到了更新，否则的话，必须等到线程空闲时 UI 才会更新。这个从后面的示例中，可以得到解释。

​ 在 Google 上查询 DoEvents，有帖子提到 DoEvents，是邪恶的，它可能会导致其他的问题，比较我遇到过的就是在一个非模态窗口中，调用 DoEvents 导致整个进程都卡死了，大概是发生了线程锁死的问题。

MSDN的官方说明

当运行Windows窗体时，会创建新窗体，然后窗体将等待处理各种事件，而窗体每次处理事件时，都会处理完与该事件关联的所有代码，才会去处理其他事件，这时其他事件将会在队列中等待。因此，在处理事件时，程序并不进行响应，当然也不会进行窗口重绘操作。但是 System.Windows.Forms.Application.DoEvents() 方法可以使窗体处理其他事件，所以窗体能够进行重绘。

在我们的实例中，当点击“Start Countdown”按钮后便开始执行与按钮关联的循环代码，在这个过程中，窗体并不会进行重绘，所以无法实时地显示当前的倒计时。使用 System.Windows.Forms.Application.DoEvents() 方法后，窗体便可以正常进行重绘进而实时地显示当前的倒计时。

使用 System.Windows.Forms.Application.DoEvents() 方法可以实时重绘窗体而避免出现画面假死的现象，但是使用 System.Windows.Forms.Application.DoEvents() 方法会带来一些效率上的问题，要是对效率有要求时，就需要考虑其他方式，如使用异步操作来完成耗时的循环操作。

后台刷新

​ 不直接设置进度，而是在 Dispatcher 的回调函数中设置进度。

namespace ProgressBarDemo.Views
{
    public class DispatcherBackgroundInvokeUserControl : ProgressBarUserControl
    {
        public override void Start()
        {
            for (int i = 0; i <= 100; i++)
            {
                progressbar.Dispatcher.Invoke(() => Progress = i, System.Windows.Threading.DispatcherPriority.Background);
                //progressbar.Dispatcher.Invoke(() => progressbar.Value = i, System.Windows.Threading.DispatcherPriority.Background);
                System.Threading.Thread.Sleep(30);
            }
            progressbar.Dispatcher.Invoke(() => { }, System.Windows.Threading.DispatcherPriority.Background);
        }
    }
}

​ for 循环外还需要再调用一次，否则最后设置的进度不会更新。

BackgroundWorker

​ 这种方式其实就是在子线程中处理任务，并把进度或其他的状态信息通过消息的方式返回给主线程，然后由主线程去更新主线程中的窗口。

using System.ComponentModel;

namespace ProgressBarDemo.Views
{
    public class BackgroundWorkerUserControl : ProgressBarUserControl
    {
        private BackgroundWorker Worker = null;
        public override void Start()
        {
            Worker = new BackgroundWorker();
            Worker.DoWork += DoWork;
            Worker.ProgressChanged += Worker_ProgressChanged;
            Worker.WorkerReportsProgress = true;
            Worker.RunWorkerAsync();
        }

        private void Worker_ProgressChanged(object sender, ProgressChangedEventArgs e)
        {
            Progress = e.ProgressPercentage;
        }

        private void DoWork(object sender, DoWorkEventArgs e)
        {
            for (int i = 0; i <= 100; i++)
            {
                Worker.ReportProgress(i);
                System.Threading.Thread.Sleep(30);
            }
        }
    }
}

async/awsit/Progress<>

​ 这种方式必须要在 .net 4.5 以上的版本才能使用，原理类似于 BackgroundWorker，只不过使用 C# 的语法糖，简化了使用的操作。推荐。

using System;
using System.Threading.Tasks;

namespace ProgressBarDemo.Views
{
    public class AsyncUserControl : ProgressBarUserControl
    {
        public async override void Start()
        {
            var progress = new Progress<double>(value => Progress = value);
            await Task.Run(() =>
            {
                for (int i = 0; i <= 100; i++)
                {
                    ((IProgress<double>)progress).Report(i);
                    System.Threading.Thread.Sleep(30);
                }
            });
        }
    }
}

多线程控件

​ 以上的方法中，进度条控件都是与上级控件（窗口）在同一线程中，WPF 可能让窗口中的控件与处理不同的线程模型中。这样的好处是，控件有自己独立的线程，而这个线程不用来处理任务，只用来处理控件的消息、刷新等。所以，在实现上很麻烦，但实现完成后使用就很简单，很自然。

AsyncProgressBar

using ProgressBarDemo.Threading;
using System;
using System.Windows;

namespace ProgressBarDemo.Controls
{
    public class AsyncProgressBar : DispatcherElementContainer<System.Windows.Controls.ProgressBar>
    {
        public AsyncProgressBar()
        {
            Loaded += AsyncProgressBar_Loaded;
        }

        private System.Windows.Controls.ProgressBar ProgressBar => Child;
        private async void AsyncProgressBar_Loaded(object sender, System.Windows.RoutedEventArgs e)
        {
            Loaded -= AsyncProgressBar_Loaded;
            await this.SetChildAsync();
        }

        public double Value
        {
            get
            {
                if (ProgressBar != null)
                {
                    return ProgressBar.Dispatcher.Invoke(() => ProgressBar.Value);
                }
                return 0;
            }
            set
            {
                SetValue(ValueProperty, value);
            }
        }

        // Using a DependencyProperty as the backing store for Value.  This enables animation, styling, binding, etc...
        public static readonly DependencyProperty ValueProperty =
            DependencyProperty.Register("Value", typeof(double), typeof(AsyncProgressBar), new PropertyMetadata(0.0,new PropertyChangedCallback(OnValueChanged)));

        private static void OnValueChanged(DependencyObject d, DependencyPropertyChangedEventArgs e)
        {
            (d as AsyncProgressBar).OnValueChanged( Convert.ToDouble(e.NewValue));
        }
        private void OnValueChanged(double value)
        {
            if (ProgressBar != null)
            {
                ProgressBar.Dispatcher.Invoke(() => ProgressBar.Value = value);
            }
        }
    }
}

效果

​ 如果可以使用子线程处理数据，最好使用子线程，.Net 4.5 以上使用 async/await/Progress，如果不是则使用 BackgroundWorker。对于一些不支持多线程的操作，使用后台刷新的方式。多线程的控件实现比较复杂，当在本实例实现的多线程控件使用在 TabControl 之中时，由于多次创建，会导致程序的崩溃。

放弃幻想

​ 前几天想做一个功能，用户按下按钮之后，调用一个子进程处理数据，同时，窗口上显示处理进度，处理完成才能进行下一步的操作。简单点说就是既要刷新窗口，又要卡线程。采用隔离的方式，使用 BackgroundWorker 在子线程中调用子进程，并把进度返回到当前窗口，并更新进度。如果是按照人的思维，我的执行停在这一行，然后去处理其他的事情，等这一行完成之后，再回来往下处理。这在人的思维比较直接容易，但是对于电脑则不行，因为电脑并不能跳脱然后再回来，最后使用的方法是，使用 DoEvents 处理序列中的事件，并使用 Sleep 来阻塞进程，多次执行并检测，直到子进程执行完成为止。
